Group 2 Developer Documentation

Product Info

This project is an inventory management system for the HRDC Warming Hut in Bozeman, Montana. Our goal for this project is to: Create a database to manage the quantities of supplies the Warming Hut uses on a daily basis Alert Warming Hut employees when items reach a certain percentage of inventory.

Developer Documentation

How to obtain source code

Clone the GitHub repository using: git clone

https://github.com/423S25/project2.git

Alternatively we will also do releases on github that include a tar file of the source code.

Planned Directory Structure

The repository is structured into three main folders, the frontend, backend, and ingress folders.

  • Frontend: The frontend folder contains all of the code for the user interface of the application. All the code is found in the sre directory, the src directory will have 4 main folder. A components folder that contains all of the react components or UI components that will be shared across the application. A pages folder that contains each of the seperate pages such as the homepage or user login page. The api folder that stores all the api endpoints we are going to be hitting from the backend. Finally, is the hook folder that contains all react hooks for reactivity that will be used across the app. There is also the public directory that contains all static files such as images and the favicon for the app. There is also the package.json file that defines all packages and the vite.config file that contains settings for our application.
  • Backend: The backend folder is relativley simple. There is a requirements.txt file that contains all the required python packages. The entry point for the code is the manage.py folder and all other django backend code can be found in the backend and management folder. The management folder will contain all our endpoints in the views.py folder. As well the database schema can be found in the models.py folder.

How to Build

Before you run the software, you have to make sure that you have docker installed on your computer.

  • Linux: Once you have that, pull all the changes code from the github repository and then run the “make build” command in the terminal. Finally, you can run “make dev” after. As well if a .env file does not exist use "touch .env"
  • Windows: With windows Makefile tools may not be available so run instead "docker-compose --env-file .env -f compose.yaml build" to build and "docker-compose --env-file .env -f compose.yaml up". As well if a .env file does not exist use "New-Item .env".
  • Note: Older versions of docker use "docker compose" instead of "docker-compose" so if docker-compose is not a valid command use "docker compose" instead either in the makefile or as a full command.

How to Test

We will have some of the group developers test the code and its functionality after each devolper edits or adds code. Future tests can be implemented using the standard Python testing library where a simple rule in our make file can be created that will run all tests. As django also has a testing command that will run any file that starts with a test. So a simple command like “make test” will test all backend features. Currently testing can be done on the delete requests in the backound based on the tests found in test.py in the management folder of the backend. For the frontend we will have to develop the idea but will likely use selenium or Jest and similarily add a rule to our makefile.

Automated Build and Test

Currently do not have an Automated Test feature, this will change as we get further along. However, NorthFlank which is our hosting service allows for a staging service. Here we can define an automated script that will run when pushed to staging from our Dockerfile that will fail the build if the tests do not pass.

How to release a version

Currently a release works through a CI/CD action where any new changes to the main branch of the repo will trigger a new build on our server on NorthFlank automatically.

Bug tracker

All bugs will be tracked using the issues page on our github repo. All issues thus can be found here https://github.com/423S25/project2/issues. Similar to users, the issue will include expected behavior, current behavior, steps to reproduce, context (Environment), and detailed description but also include possible solution and possible implementation.
